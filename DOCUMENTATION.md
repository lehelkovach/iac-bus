## System Overview
The Inter-Agent Communication Bus (IAC Bus) is a small Flask application that
accepts messages over HTTP and keeps them in memory. It is designed for quick,
lightweight coordination between agents without requiring external storage.

## Message Model
Each message stored in the bus has the following fields:

| Field | Type | Description |
| --- | --- | --- |
| `id` | string | UUID hex string generated by the server |
| `ts` | number | Unix epoch time in seconds (float) |
| `channel` | string | Channel name for grouping messages |
| `sender` | string | Sender identifier |
| `message` | string | Message payload |

## Authentication
If `BUS_API_TOKEN` is set, all endpoints except `/health` require a bearer
token header:

```
Authorization: Bearer <token>
```

If `BUS_API_TOKEN` is empty, the API accepts requests without authentication.

## API Reference

### POST /bus/messages
Create a new message.

Request body (JSON):
- `channel` (string, optional, default: `"default"`)
- `sender` (string, optional, default: `"agent"`)
- `message` (string, required)

Response:
- `201 Created` on success
- `400 Bad Request` if `message` is missing
- `401 Unauthorized` if the token is required and missing/invalid

Example:
```bash
curl -X POST "http://127.0.0.1:8091/bus/messages" \
  -H "Authorization: Bearer $BUS_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"channel":"ops","sender":"agent-a","message":"ready"}'
```

### GET /bus/messages
Fetch messages.

Query parameters:
- `channel` (string, optional) - filter by channel
- `since_id` (string, optional) - return messages after this ID
- `limit` (int, optional, default: `50`, max: `200`)

Response:
- `200 OK` with `{"messages":[...]}`.

Example:
```bash
curl "http://127.0.0.1:8091/bus/messages?channel=ops&limit=100" \
  -H "Authorization: Bearer $BUS_API_TOKEN"
```

### GET /health
Returns basic health status. This endpoint does not require authentication.

Example:
```bash
curl "http://127.0.0.1:8091/health"
```

## Configuration

| Variable | Default | Purpose |
| --- | --- | --- |
| `BUS_HOST` | `0.0.0.0` | Bind address |
| `BUS_PORT` | `8091` | Listen port |
| `BUS_API_TOKEN` | empty | Bearer token |
| `BUS_MAX_MESSAGES` | `500` | Max retained messages |
| `BUS_RETENTION_SECONDS` | `3600` | Message retention window |
| `BUS_LOG_LEVEL` | `INFO` | Log level |

## Runtime Behavior
- Messages are pruned on insert and on read.
- Retention is enforced by time (`BUS_RETENTION_SECONDS`) and size
  (`BUS_MAX_MESSAGES`).
- `limit` is capped at `200`.
- `since_id` returns messages strictly after the matching ID if found; if the
  ID is not present, all messages for the query are returned.

## Deployment

### Local Run
```bash
python3 -m venv venv
./venv/bin/pip install -r requirements.txt
BUS_API_TOKEN=devtoken ./venv/bin/python server.py
```

### Production (systemd)
```bash
sudo ./deploy.sh
sudo systemctl status iac-bus.service
```

`deploy.sh` installs the service in `/opt/iac-bus` and writes configuration to
`/etc/iac-bus/iac-bus.env`.

### Gunicorn (optional)
The repository includes `gunicorn` in `requirements.txt`. You can run:

```bash
BUS_API_TOKEN=devtoken ./venv/bin/gunicorn -b 0.0.0.0:8091 server:app
```

## Operations
- Logs are emitted to stdout using the configured `BUS_LOG_LEVEL`.
- The systemd unit restarts on failure.
- The service runs as `root` by default in the provided unit file.

## Scripts

### Hotfix (local VM)
Pulls a tarball from GitHub and redeploys.

```bash
sudo ./scripts/hotfix-pull.sh
```

Environment variables:
- `REPO_OWNER` (default: `lehelkovach`)
- `REPO_NAME` (default: `iac-bus`)
- `REF` (default: `main`)
- `INSTALL_DIR` (default: `/opt/iac-bus`)
- `TMP_DIR` (default: `/tmp/iac-bus-hotfix`)

### Hotfix (remote VM)
Runs the hotfix procedure over SSH.

```bash
./scripts/hotfix-remote.sh ubuntu@<IP> ~/.ssh/your_key
```

Environment variables:
- `REPO_OWNER` (default: `lehelkovach`)
- `REPO_NAME` (default: `iac-bus`)
- `REF` (default: `main`)

### Spawn Cursor Agents
Spawns Cursor Cloud agents and announces each agent on the bus.

```bash
export CURSOR_API_KEY="key_xxx..."
python3 scripts/spawn-cursor-agents.py \
  --cursor-endpoint "https://api.cursor.com/v0/agents" \
  --count 2 \
  --payload '{"name":"agent-a"}' \
  --bus-url "http://127.0.0.1:8091" \
  --bus-token "$BUS_API_TOKEN"
```

Notes:
- Cursor API uses Basic Auth with the key as the username and blank password.
- `--payload` is passed directly to the Cursor API.

### Handoff: Terminate Cursor Agents
Terminate agents by ID, file, or bus announcements.

```bash
export CURSOR_API_KEY="key_xxx..."
python3 scripts/handoff-terminate-agents.py \
  --cursor-endpoint "https://api.cursor.com/v0/agents" \
  --agent-ids "agent-id-1,agent-id-2"
```

```bash
python3 scripts/handoff-terminate-agents.py \
  --cursor-endpoint "https://api.cursor.com/v0/agents" \
  --agent-ids-file /path/to/handoff_notes.md
```

```bash
python3 scripts/handoff-terminate-agents.py \
  --cursor-endpoint "https://api.cursor.com/v0/agents" \
  --bus-url "http://127.0.0.1:8091" \
  --bus-token "$BUS_API_TOKEN"
```

## Troubleshooting
- `401 Unauthorized`: check `BUS_API_TOKEN` and the `Authorization` header.
- `400 message required`: ensure the POST body includes a non-empty `message`.
- Empty message list: verify the `channel`, `since_id`, and retention settings.

## Limitations
- Messages are stored only in memory; restarts clear all data.
- If you run multiple processes (for example, gunicorn with multiple workers),
  each process maintains its own message list.
- There is no built-in persistence or fine-grained access control.
